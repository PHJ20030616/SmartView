## 2. 问题与修复方案

### 2.1 P1：默认 JWT 密钥可预测

**影响**

- `application.yml` 中存在可预测的默认 JWT 密钥。
- 部署环境未配置 `JWT_SECRET` 时，应用仍可启动并使用公开密钥签发和验证令牌。
- 攻击者可以伪造合法 JWT，直接绕过身份认证。

**根因**

- 为方便本地启动，在默认配置中给环境变量设置了固定回退值。
- 默认配置同时被生产环境继承，未明确区分测试密钥与部署密钥。

**修复方案**

1. 移除默认配置中的固定密钥回退值，仅允许从 `JWT_SECRET` 注入签名密钥。
2. 在 JWT 配置绑定阶段校验密钥非空及长度满足签名算法要求，使缺失或弱密钥在应用启动阶段立即失败。
3. 测试环境继续通过 `application-test.yml` 提供独立固定密钥，确保测试可重复且不依赖开发机环境变量。
4. 添加必要中文注释，说明默认环境禁止密钥回退的安全原因，且不得记录密钥内容。

**验证方式**

- 未设置 `JWT_SECRET` 时，默认配置启动失败并给出明确的配置错误。
- 测试配置下应用上下文正常启动。
- 合法密钥签发和解析 JWT 的原有测试继续通过。

### 2.2 P2：未知用户登录未执行 BCrypt，存在时序侧信道

**影响**

- 用户名不存在时直接返回，用户名存在但密码错误时会执行 BCrypt。
- 两条路径耗时差异明显，攻击者可通过大量采样推测用户名是否存在。
- 虽然接口文案一致，仍可能通过时序泄露账号存在性。

**根因**

- 登录逻辑只在查到用户后调用 `passwordEncoder.matches`。
- 未为不存在用户的分支设置等价的密码哈希校验成本。

**修复方案**

1. 准备一个仅用于时序平衡的固定 BCrypt 哈希，不对应任何真实账号。
2. 用户不存在时仍调用一次 `passwordEncoder.matches`，使不存在用户和密码错误路径都承担 BCrypt 校验成本。
3. 两种失败场景继续返回相同状态码和中文错误信息，不暴露具体失败原因。
4. 添加中文安全注释，说明该次无业务结果的 BCrypt 校验用于降低用户名枚举风险。

**验证方式**

- 增加测试验证未知用户分支确实调用密码匹配。
- 未知用户与错误密码继续返回相同响应状态和错误码。
- 日志及响应均不包含用户名是否存在、原始密码或密码哈希。

### 2.3 P2：字段在 `trim()` 前完成校验，规范化后可能违反契约

**影响**

- 用户名、昵称等字段可能在带前后空白时先通过 DTO 校验，随后 `trim()` 后长度不足或变为空值。
- 数据库存储值可能不满足 OpenAPI 对最小长度、非空等约束。
- `"user"`、`" user "` 等输入的唯一性行为可能不一致，产生绕过预检查或错误状态码的问题。

**根因**

- Bean Validation 在 Controller 入参绑定阶段执行，而字符串规范化发生在 Service 内。
- 规范化后没有再次按照契约约束执行完整校验。

**修复方案**

1. 明确用户名、昵称、邮箱和手机号等可规范化字段的处理规则，并在进入持久化及唯一性检查前统一去除前后空白。
2. 对规范化后的值执行与 OpenAPI 契约一致的非空、长度及格式校验。
3. 唯一性预检查和最终保存使用同一份规范化值，确保空白变体不能绕过冲突检测。
4. 对可选字段区分 `null` 与空白字符串，避免把无意义空字符串写入数据库。
5. 对复杂的规范化顺序和唯一性约束添加必要中文注释。

**验证方式**

- 增加纯空白、前后空白、规范化后长度不足等测试。
- 增加 `"user"` 与 `" user "` 的冲突测试。
- 验证数据库最终保存的是规范化后的值，并始终满足契约。

### 2.4 P2：登录接口 OpenAPI 契约遗漏实际 403 和 422 响应

**影响**

- 非 ACTIVE 用户登录时实际返回 403，但契约未声明。
- 规范化或业务校验失败可能返回 422，但契约未声明。
- 前端生成客户端、接口文档和契约测试无法完整表达运行时行为。

**根因**

- 初始契约只覆盖了成功、常规参数错误和认证失败场景。
- 实现过程中新增的账号状态和业务校验分支未同步完整反映到契约。

**修复方案**

1. 先修改 `contracts/web-api/openapi.yaml`，为登录接口补充实际可能出现的 403 和 422 响应。
2. 复用项目统一错误响应模型，保持错误码、HTTP 状态和运行时实现一致。
3. 检查注册与当前用户接口的响应声明，避免同类遗漏。
4. 重新执行 OpenAPI 代码生成和编译，禁止手动修改生成目录。

**验证方式**

- OpenAPI 校验和代码生成成功。
- 集成测试分别覆盖非 ACTIVE 用户的 403 和规范化校验失败的 422。
- Controller、异常处理器和契约声明的状态码保持一致。

### 2.5 P3：无效 JWT 会阻断公开健康检查和接口文档

**影响**

- 请求公开的健康检查、OpenAPI 或 Swagger UI 时，如果携带过期或伪造 JWT，过滤器可能提前返回 401。
- 公开端点的可用性依赖客户端是否清理了旧令牌，影响监控、文档访问和排障。

**根因**

- JWT 过滤器目前只跳过注册和登录路径。
- Security 中已配置的其他公开路径没有与过滤器的跳过规则保持一致。

**修复方案**

1. 将公开端点匹配规则集中复用，避免 Security 授权配置和 JWT 过滤器维护两份不一致的路径列表。
2. 对注册、登录、健康检查、OpenAPI 和 Swagger UI 等明确公开端点跳过 JWT 认证过滤。
3. 受保护接口携带无效 JWT 时仍必须返回统一 401。
4. 添加中文注释说明“公开端点忽略无效令牌”只适用于明确白名单，不得扩大到受保护 API。

**验证方式**

- 公开端点在无令牌、合法令牌和无效令牌下均按公开规则正常访问。
- `/api/users/me` 在无令牌、伪造令牌和过期令牌下均返回 401。
- 未在公开白名单中的其他接口仍保持认证要求。

### 2.6 P3：关键安全与并发边界缺少集成测试

**影响**

- 过期 JWT 当前只在 `JwtService` 单元测试中验证，未覆盖过滤器到 HTTP 401 的完整链路。
- 并发注册可能同时通过应用层预检查，数据库唯一约束异常转换是否稳定缺少验证。
- 大小写变体、前后空白变体和规范化冲突缺少明确回归保障。

**根因**

- 首轮集成测试聚焦核心验收路径，没有覆盖数据库竞争和全部输入规范化边界。

**修复方案**

1. 增加过期 JWT 经 `JwtAuthenticationFilter` 访问 `/api/users/me` 并返回统一 401 的集成测试。
2. 增加同一唯一字段并发注册测试，断言仅一个请求成功、另一个返回 409，数据库只保留一条记录。
3. 根据数据库和既有业务规则补充用户名大小写变体测试，明确并固定当前预期行为。
4. 增加用户名、邮箱、手机号前后空白导致的规范化冲突测试。
5. 验证失败响应不泄露数据库异常、JWT 底层异常或敏感字段。

**验证方式**

- 新增边界测试全部通过，且能在修复前稳定暴露相应问题。
- 全量测试无回归。
- 并发用例重复运行时结果稳定，不依赖执行顺序。

## 7. 第二轮审查问题与修复方案

### 7.1 P2：邮箱契约长度与数据库字段不一致

**影响**

- OpenAPI 中邮箱只有格式约束，没有数据库 `varchar(100)` 对应的最大长度约束。
- 超过 100 字符且格式合法的邮箱可能通过生成 DTO 校验，在持久化时触发数据库异常并返回 500。

**修复方案**

1. 先在 `contracts/web-api/openapi.yaml` 为注册邮箱增加 `maxLength: 100`。
2. 通过 Maven 重新生成 DTO，不手工修改生成目录。
3. 增加超长合法格式邮箱的集成测试，断言返回 422 且不写入数据库。

**验证方式**

- 生成的 `RegisterRequest` 包含邮箱 `@Size(max = 100)` 约束。
- 超长邮箱请求返回统一中文 422 响应。

### 7.2 P2：空请求体和畸形 JSON 未映射为稳定的 400 响应

**影响**

- `@RequestBody` 在进入业务服务前反序列化失败，当前可能落入通用异常处理并返回 500。
- 注册契约已声明 400，但运行时实现可能不一致；登录契约尚未声明 400。

**修复方案**

1. 先为登录接口补充 400 契约响应。
2. 在全局异常处理器中显式处理请求体缺失或 JSON 解析失败异常，返回统一 400 和中文提示。
3. 增加注册、登录空请求体与畸形 JSON 集成测试，覆盖反序列化失败链路。

**验证方式**

- 注册、登录的空请求体和畸形 JSON 均返回 HTTP 400。
- 响应使用统一错误结构，不暴露 Jackson 或 Spring 底层异常。

## 10. 第三轮独立审查问题

### 10.1 P1：登录成功更新会覆盖管理员并发禁用状态

**影响**

- 登录流程读取 `ACTIVE` 用户后使用 `updateById` 写回完整旧实体。
- 若管理员在密码校验期间禁用账号，旧实体可能把最新状态重新覆盖为 `ACTIVE`，并继续签发 JWT。

**修复**

- 在 `UserMapper` 增加仅更新 `last_login_at` 的原子 SQL，并同时限定用户 ID、`ACTIVE` 状态和
  `deleted = 0`。
- 条件更新返回 0 时终止登录，返回 HTTP 403 和中文提示，且不调用 JWT 签发。
- 添加中文安全注释，说明条件更新用于防止旧实体覆盖并发账号状态。

### 10.2 P2：BCrypt 72 字节限制可能导致超长多字节密码返回 500

**影响**

- 原契约允许最多 128 个字符，但 BCrypt 只接受最多 72 字节。
- 中文等 UTF-8 多字节密码可能字符数不多却超过 72 字节，并在编码或比对阶段抛出异常。

**修复**

- 将注册和登录密码的 OpenAPI `maxLength` 调整为 72。
- 服务层在调用 BCrypt 前按 UTF-8 字节数校验，超过 72 字节统一返回 HTTP 422。
- 注册超限时验证数据库无新增用户记录。

## 12. 第四轮独立审查问题

### 12.1 P1：JWT 过滤器会把下游参数异常误写为 401

**影响**

- `JwtAuthenticationFilter` 原先在同一个 `try` 块中完成 JWT 解析、认证上下文创建和
  `filterChain.doFilter` 调用。
- 下游控制器、拦截器或过滤器抛出 `IllegalArgumentException` 时，会被误认为 JWT 格式错误，
  原始异常被吞掉并返回“登录状态无效”的 HTTP 401。

**修复**

- 新增过滤器回归测试，使用有效 JWT 和有效用户后主动让下游链抛出参数异常。
- 仅捕获 `JwtService.parseAccessToken` 抛出的 JWT 解析异常。
- 将数据库读取、认证上下文创建和下游过滤器链调用移出 JWT 异常捕获范围。
- 添加必要中文注释，明确下游异常必须交给 MVC 或全局异常处理链。

### 12.2 P3：生成请求 DTO 的 `toString()` 仍包含密码

**处理**

- 注册和登录密码字段已在 OpenAPI 契约中标记为 `writeOnly: true`，明确密码只允许作为请求输入。
- OpenAPI Generator 7.14.0 当前仍会在生成 DTO 的 `toString()` 中输出密码字段；生成目录按项目规则
  不允许手工修改。
- 当前认证代码不记录请求 DTO，故该问题不阻塞 Task 2.2 验收；后续可通过生成器模板统一脱敏。
