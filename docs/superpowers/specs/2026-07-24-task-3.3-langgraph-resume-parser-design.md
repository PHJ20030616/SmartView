# Task 3.3 LangGraph 简历解析设计

## 1. 目标与范围

在 `smartview-ai` 中实现 `POST /api/v1/resume/parse`，接收 Spring Boot 提供的 PDF 预签名 URL，完成 PDF 文本提取、PaddleOCR 中文兜底、DeepSeek 结构化解析，并返回基本信息、教育、工作、项目、技能和原始文本。

本任务同时提供可由异步 worker 复用的解析入口，但不负责 Spring Boot 侧的 MQ 结果消费和数据库写入。

## 2. 技术方案

采用 LangGraph `StateGraph` 构建受控工作流。文件下载、PDF 解析、文本质量判断、OCR 和结果校验均为确定性节点；DeepSeek 只负责从原始文本提取结构化字段，不允许模型自行选择或跳过处理步骤。

流程如下：

```text
START
  -> 下载并校验 PDF
  -> 逐页提取原生文本
  -> 判断是否存在低质量页面
      -> 否：合并文本
      -> 是：仅对低质量页面执行 PaddleOCR
  -> 合并和清洗原始文本
  -> DeepSeek JSON 结构化解析
  -> Pydantic 校验
  -> END
```

短时、单次的简历解析不启用 LangGraph checkpointer。任务状态、重试次数和幂等控制继续由 Spring Boot、RabbitMQ 与数据库负责，避免出现两套权威状态。

## 3. 契约与数据模型

先更新 `contracts/ai-api/openapi.yaml`，然后实现对应 Pydantic 模型。

### 请求

- `fileUrl`：MinIO 预签名 URL。
- `mimeType`：当前仅接受 `application/pdf`。
- `traceId`：Spring Boot 传入的 UUID 链路标识。

### 响应

- `success`
- `candidateName`
- `contactInfo`
- `education`
- `workExperience`
- `projectExperience`
- `skills`
- `rawText`
- `errorMessage`

教育、工作和项目经历使用明确的嵌套 Pydantic 模型。LLM 缺失的可选字段保留为空，列表字段默认返回空数组。成功响应必须包含 `rawText`；失败通过统一异常响应返回中文可读消息，不返回半结构化结果。

## 4. PDF 与 OCR

使用 PyMuPDF 从下载后的 PDF 字节逐页提取文本，并保留页序。文本质量判断基于去除空白后的有效字符数，不仅判断字符串是否为空，避免图片水印、页码等少量噪声被误认为有效正文。

仅对原生文本低于阈值的页面渲染高分辨率位图并调用 PaddleOCR，文本型页面不执行 OCR。这样可兼容纯文本 PDF、扫描 PDF 和混合 PDF，同时控制 OCR 延迟。

PaddleOCR 使用延迟初始化，避免健康检查和纯文本 PDF 请求加载大型模型。OCR 依赖不可用、模型初始化失败或识别结果为空时，返回中文业务错误，并记录详细服务端日志。

## 5. DeepSeek 接入

使用 `httpx.AsyncClient` 调用 DeepSeek OpenAI 兼容接口，不引入额外供应商 SDK。请求使用非流式 `/chat/completions` 和 JSON Output：

```json
{
  "response_format": {"type": "json_object"}
}
```

系统提示词明确给出字段语义、禁止编造、日期原样保留和 JSON 输出要求。模型输出先解析为 JSON，再由 `ResumeStructuredData` 校验；不合法时进行一次带校验错误信息的修复调用，第二次仍失败则返回可读错误。

配置项：

- `DEEPSEEK_API_KEY`：必填密钥，只允许通过环境变量或未跟踪的 `.env` 设置。
- `DEEPSEEK_BASE_URL`：默认 `https://api.deepseek.com`。
- `DEEPSEEK_MODEL`：模型名，可在配置文件中覆盖。
- `DEEPSEEK_TIMEOUT_SECONDS`：调用超时。
- `DEEPSEEK_MAX_TOKENS`：最大输出 token。
- `DEEPSEEK_TEMPERATURE`：默认使用低温度以提高结构化稳定性。

仓库中的示例配置只写变量名和非敏感默认值，禁止提交真实 API Key。用户提供的本机密钥写入根目录 `.env`，该文件已被 `.gitignore` 忽略。

## 6. 模块边界

- `app/schemas/resume.py`：请求、响应、嵌套经历模型和 LangGraph 状态。
- `app/services/resume_parser.py`：下载、PDF 提取、OCR、DeepSeek 调用及文本质量工具。
- `app/graphs/resume_graph.py`：节点定义、条件路由和图编译。
- `app/api/v1/resume.py`：HTTP 接口、依赖获取和错误转换。
- `app/workers/resume_worker.py`：复用解析图的异步任务处理函数。
- `app/core/config.py`：DeepSeek、下载、PDF 和 OCR 配置。

API 与 worker 必须调用相同的图入口，禁止复制解析流程。

## 7. 错误处理

对外错误统一使用现有 `AppError` 和全局异常处理器：

- 非 PDF MIME：`UNSUPPORTED_RESUME_TYPE`，HTTP 400。
- 下载失败或预签名 URL 失效：`RESUME_DOWNLOAD_FAILED`，HTTP 400。
- PDF 损坏或无法打开：`INVALID_RESUME_PDF`，HTTP 400。
- PDF 与 OCR 均无有效文本：`RESUME_TEXT_UNAVAILABLE`，HTTP 422。
- OCR 服务不可用：`RESUME_OCR_FAILED`，HTTP 500。
- DeepSeek 配置缺失：`LLM_NOT_CONFIGURED`，HTTP 500。
- DeepSeek 超时或调用失败：`LLM_REQUEST_FAILED`，HTTP 502。
- LLM JSON 不合法：`RESUME_STRUCTURE_FAILED`，HTTP 502。

错误响应只暴露中文可读信息、稳定错误码和 `traceId`，不暴露密钥、预签名 URL、模型原始异常或堆栈。

## 8. 测试与验收

采用测试驱动开发，覆盖：

1. 文本型中文 PDF 能提取原文并返回结构化简历。
2. 原生文本有效时不会初始化或调用 PaddleOCR。
3. 扫描页或低质量页进入 OCR 条件分支。
4. 混合 PDF 仅对低质量页执行 OCR，并按页序合并。
5. DeepSeek JSON 返回经过 Pydantic 校验。
6. 首次 JSON 非法时执行一次修复，第二次失败返回可读错误。
7. MIME 非法、下载失败、损坏 PDF、空文本和 LLM 超时均返回稳定错误。
8. FastAPI 生成的 `/api/v1/resume/parse` OpenAPI 与 `contracts/ai-api/openapi.yaml` 的路径、operationId、请求和响应核心字段一致。
9. worker 与 API 复用同一图入口。

测试默认模拟 DeepSeek 和 PaddleOCR，避免依赖网络、余额和大型 OCR 模型。另使用测试代码生成一份真实中文文本型 PDF，验证 PyMuPDF 提取链路。

## 9. 安全与运维

- 日志不得记录 API Key、完整预签名 URL 或完整简历原文。
- 下载设置连接、读取和总超时，并限制最大文件大小。
- PDF 页数、渲染分辨率和文本长度设置上限，防止资源耗尽。
- DeepSeek 请求设置超时，worker 失败交由既有 MQ 重试策略处理。
- 所有关键日志携带 `traceId`，便于跨服务排查。
